## Cluster 1 — Data corruption (the numbers are fiction)

- **R1 (C)** — Decay compounds per cycle. `lib/consolidation.mjs:115-141`: `salience *= 0.5^(days_since_recall/14)` anchored to `last_recalled||last_seen` (not last decay application), applied every 30-min scheduler cycle, overwriting salience. Decay meant to take weeks completes in hours. Runtime: 961 rows in `entities_archived` vs 110 live entities; "Decayed: 110" (all survivors) every cycle.
- **R2 (H)** — Reinforcement non-idempotent. `lib/consolidation.mjs:227-249`: qualifying co-occurrence pairs get `mention_count += 1, salience += 0.05` every cycle, forever, no already-reinforced record. Top entities at mention_count ~148 = scheduler cycles, not mentions. R1+R2 hold a fake equilibrium (~0.158 salience).
- **R3 (C)** — Daemon tick re-entrancy. `workspace-bin/memory-daemon.mjs:1566`: `setInterval(async () => { await tick(); }, 30_000)` with no in-flight guard; ticks measured 30-40s+ (LLM extraction). Throttle state read at tick start, persisted at end → overlapping ticks re-run the same flush/import/synthesis concurrently. Duplicate extraction, racing MEMORY.md writes, Phase 2 concurrent with Phase 0. Found independently by two reviewers.
- **R4 (H)** — Every flush re-extracts the same tail. `runFlush`→`storeExtractionResult` has no content-hash/turn-range dedup; interval (30min), IDLE→ENDED, session-switch and NATS triggers all re-process unchanged tails → duplicate mention rows (third mention_count inflator).
- **R5 (M)** — turn_index off-by-one. `lib/pre-compression-flush.mjs:393` stamps `messageCount`; turns are 0-based (max `messageCount-1`). Every mention points at a nonexistent turn; turn-grain filtering can never match.

## Cluster 4 — Daemon lifecycle

- **R15 (H)** — Shutdown doesn't fence in-flight work: `shutdown()` closes DB handles + drains NATS without awaiting an in-flight tick; actual exit deferred to next 30s interval fire (past launchd timeout). Runtime: last daemon exit status -9 (SIGKILL); native `mutex lock failed` abort in `.err`.
- **R16 (M)** — NATS init is one-shot at startup: event log, watcher, extraction trigger all init in one try block; if NATS is down at boot they stay dead until manual restart. Health probes (`healthProbeTimer`) also wrongly inside the NATS block — SQLite probes have no NATS dependency.
- **R17 (M)** — IDLE→ENDED cleanup uses `findCurrentJsonl()` (`memory-daemon.mjs:1117,1180`) not `findJsonlBySessionId` — a session ended by a new session appearing can flush/archive the wrong transcript. ACTIVE→ENDED path does it right.
